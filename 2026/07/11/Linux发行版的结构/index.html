问:根文件系统有哪些?

答:

名称本质生命周期
rootfs内核内置 tmpfs整个运行期间都存在（被覆盖）
initrdRAM disk 块设备仅早期启动阶段（已过时）
initramfscpio + tmpfs仅早期启动阶段，现代主流
real rootfs磁盘/网络文件系统switch_root 后接管 /

问: 为什么要有initramfs与real rootfs之分?
有了initrd.img之后,Linux已经可以交互了,将initramfs与real rootfs合并的话,管理不是更简单吗?

答: 关键在于ext4等驱动的问题
如果要挂载ext4格式的根文件系统,那么必须先有ext4驱动,那ext4驱动去哪里找?编译到内核中?
如果ext4驱动编译到内核中的话,那么其他场景呢?
其他文件系统驱动也编译到内核?
LUKS解密逻辑也写在内核中?
这样会导致发行版要为每种硬件维护单独的内核 -> 不可能完成
或者内核代码爆炸式增长,编译出来的内核十分臃肿

为什么PATH不是环境变量?
这实际上是由于bash在启动时,检测到PATH不存在,从而为PATH设置了默认值,但这样创建创建的变量是不会设置为环境变量的

具体源码分析:

在variables.c:552这一行:

这里的set_if_not()函数会把没有的变量设置指定值,即这里但PATH不存在时,设置其值为DEFAULT_PATH_VALUE
每一个变量都是使用SHELL_VAR结构体描述的,该结构体定义在variables.h:82:

其中,一个变量是否为环境变量由attributes决定,如果其值等于att_exported则该变量为环境变量
而att_exported定义在variables.h:104:

让我们回到set_if_not()函数,一路调试追踪,我们来到了new_shell_variable()函数内,其中设置了attributes的值的一行为variables.c:2777:

这表明默认创建的变量不是环境变量.
并且由于之后也没有代码修改它.
最终导致了PATH不是环境变量,所以bash不会将PATH传递给dpkg,导致了dpkg报错error: PATH is not set

为什么bash设置PATH默认值时,不设置其为环境变量?
这么做没有什么特别的理由,只是因为PATH不是程序运行必须的,

你也许对为什么是这样感到疑惑

这是因为Ctrl+C产生的信号是发送给当前会话的前台进程组的
而前台进程组必须先有控制终端,前台进程组的ID是记录在控制终端里的
控制终端需要程序或者用户手动绑定,内核不会自动绑定它
但bash不会去绑定控制终端,而我们又没有编写绑定相关的代码
所以导致了init的会话中没有控制终端,那么前台进程组自然也无从谈起了
结果就是Ctrl+C产生的信号没有任何进程收到,造成了Ctrl+C的无效

fg命令也类似,他通过将一个进程从后台进程组放到前台进程组中达成放到前台运行的效果,但在init的会话中,根本没有前台进程组,所以也根本不会起效果

那么那两条错误输出呢?
我想,回看他的意思的时候,你应该能能猜到一点了,没错,这是由于bash尝试获取前台进程组,由于根本没有前台进程组,所以失败了

那么为什么伪终端会帮我们处理好控制终端的问题,但init却不会呢?
这是由于init实际上不需要他们,这些东西只有在交互式终端中才会用,而init是用于初始化的,是非交互的,只是因为我们为简单地获取一个交互终端才在init中直接使用exec /bin/bash
